3.1 Architecture & Design (9.0/10)

PASS Medallion Architecture Correctly Applied
Clean separation: D_RAW (landing) → D_BRONZE (preserved). Source table untouched. CDC metadata columns properly appended (6 columns). Lineage metadata embedded in table COMMENT.
PASS Staging Pattern (Single Stream Read)
Stream data is materialized into a temporary table before MERGE — this is a critical best practice. Avoids stream offset advancement on failed MERGE and enables pre-merge metrics. Line 341-356
PASS Soft Delete Strategy
Deletes are handled as soft deletes (IS_DELETED=TRUE) preserving all historical data. This is the correct approach for a Bronze preservation layer. Line 474-480
PASS Primary Key Defined at Table Level
DTCTD_EQPMNT_ID declared as NOT NULL with PRIMARY KEY constraint. Consistent use as MERGE join key across all paths. Line 29, 86

3.2 Error Handling & Recovery (8.5/10)

PASS Stream Staleness Detection & Auto-Recovery
Proactive stale stream detection using SELECT COUNT(*) WHERE 1=0 pattern. Automatic recovery: validate base table → recreate stream with SHOW_INITIAL_ROWS → full re-sync MERGE. Lines 163-336
PASS Nested Exception Handlers
Logging failures have their own inner exception handlers to prevent cascading failures. Both STATEMENT_ERROR and WHEN OTHER are handled separately. Lines 589-639
WARN MEDIUM Recovery MERGE Could Cause Data Drift
When stream goes stale and is recreated with SHOW_INITIAL_ROWS=TRUE, the recovery MERGE re-syncs ALL rows. However, any rows deleted in the source between the stale event and recovery will NOT be marked as deleted in Bronze. Consider adding a reconciliation step post-recovery that identifies Bronze rows not present in the source and marks them IS_DELETED=TRUE.
WARN LOW No Retry Mechanism for Transient Errors
The procedure returns immediately on error. For transient network or lock contention issues, a retry loop (1-2 attempts with backoff) would improve resilience. The Snowflake Task will retry on next 5-min schedule, but this means a 5-minute delay.

3.3 CDC Logic & MERGE Correctness (9.0/10)

PASS All 4 MERGE Clauses Correctly Ordered
Clause 1: MATCHED + INSERT + IS_UPDATE=TRUE → UPDATE (standard CDC update)
Clause 2: MATCHED + DELETE + IS_UPDATE=FALSE → Soft Delete
Clause 3: MATCHED + INSERT + IS_UPDATE=FALSE → DUPLICATE_RESOLVED (edge case)
Clause 4: NOT MATCHED + INSERT → New record insertion
Order matters in MERGE — this ordering is correct per Snowflake semantics.
PASS Duplicate Insert Resolution (FIX #8)
Handles the edge case where CDC sends an INSERT for an already-existing PK (out-of-order events). Converts to UPDATE with CDC_OPERATION='DUPLICATE_RESOLVED' for auditability. Pre-merge duplicate count is tracked. Lines 383-390, 485-539
INFO DELETE + IS_UPDATE=TRUE Records Silently Dropped
When METADATA$ACTION=DELETE and METADATA$ISUPDATE=TRUE, these are the "old row" half of an UPDATE pair. The MERGE has no clause matching this condition — which is correct behavior since the INSERT+IS_UPDATE=TRUE clause handles the "new row" half. However, this means these rows in the staging table are unused. No action needed; just documenting for clarity.

3.4 Task & Scheduling (8.5/10)

PASS Intentional Omission of WHEN SYSTEM$STREAM_HAS_DATA — Correct Design Decision
Context: The source table (D_RAW.EHMS.DTQ_DTCTD_EQPMNT_BASE) is subject to external DROP & RECREATE operations by the IDMC/Oracle CDC pipeline. When the source table is dropped, the stream immediately becomes stale.

The Problem with WHEN SYSTEM$STREAM_HAS_DATA: Per Snowflake KB: "If the stream is already stale or if the STALE_AFTER is already past the current date/timestamp, then the task will fail." When SYSTEM$STREAM_HAS_DATA() is evaluated against a stale stream, it throws an error. This causes the task's WHEN clause itself to fail — the stored procedure never executes, and the built-in stale stream auto-recovery logic (Lines 163-336) is completely bypassed.

Verdict: Omitting the WHEN clause is the correct trade-off for this architecture. The SP already handles empty streams efficiently (v_staging_count = 0 early exit with NO_DATA log). The minor cost of resuming the warehouse every 5 minutes during quiet periods is vastly preferable to a permanently stuck pipeline that cannot self-heal after source table DDL events.

Cost Mitigation: The task uses the shared INFA_INGEST_WH warehouse (likely already running for other ingestion tasks), and the empty-stream path exits within seconds, so incremental cost is minimal. Line 645-651
PASS ALLOW_OVERLAPPING_EXECUTION = FALSE
Prevents concurrent runs which would cause duplicate processing. Correct for CDC pipelines.
WARN LOW No Task Failure Alerting
While execution logs are captured, there is no email/Slack notification integration for repeated failures. Consider adding an alert integration or using Snowflake's notification integrations for TASK_HISTORY monitoring.

3.5 Security & Permissions (9.0/10)

PASS EXECUTE AS CALLER — Client Approved & Architecturally Required
Client Decision: EXECUTE AS CALLER has been reviewed and explicitly approved by the client security team.

Technical Justification: The procedure operates across two databases (D_RAW for source/stream and D_BRONZE for target/monitoring). Using EXECUTE AS CALLER allows the task's owner role to provide the necessary cross-database privileges without creating a dedicated owner role with broad grants. The task is the sole caller in production, so the execution context is controlled.

Status: Accepted. No change required. Line 118
INFO Recommendation: Document Authorized Execution Roles
Since EXECUTE AS CALLER is the chosen model, it is recommended to add a comment in the procedure header listing the authorized roles (e.g., SYSADMIN, CDC_ADMIN) that are permitted to invoke this procedure. This aids future audits and onboarding.

6. Prioritized Recommendations

RESOLVED WHEN SYSTEM$STREAM_HAS_DATA — Reclassified from P0 to ACCEPTED
Previous Finding: Missing WHEN clause flagged as P0 critical.
Resolution: After analysis, this is an intentional and correct design decision. The source table is subject to external DROP & RECREATE by IDMC CDC. When the source table is dropped, the stream becomes stale. If WHEN SYSTEM$STREAM_HAS_DATA is present, the task's WHEN clause throws an error on stale streams (confirmed per Snowflake KB), which prevents the SP's built-in stale recovery from executing. Without the WHEN clause, the SP runs, detects staleness, and self-heals automatically. The trade-off of minor warehouse resume cost during quiet periods is far preferable to a permanently broken pipeline. No change required.
RESOLVED EXECUTE AS CALLER — Reclassified from P0 to ACCEPTED
Previous Finding: EXECUTE AS CALLER flagged as potentially overly permissive.
Resolution: Client security team has explicitly approved this setting. Architecturally justified: procedure spans D_RAW and D_BRONZE databases requiring cross-database privileges. Task is the sole production caller, ensuring controlled execution context. No change required.
P0 — CRITICAL (Must Fix Before Deployment)

Post-Recovery Reconciliation for Deletes

After stream stale recovery, rows deleted from the source during the stale window will not be soft-deleted in Bronze. Add a reconciliation step after recovery MERGE that identifies orphaned Bronze rows and sets IS_DELETED=TRUE.

P1 — RECOMMENDED (Address Within Sprint)

Reduce Column List Duplication

Refactor to reduce the 8x repeated 49-column lists. Options: (a) Use SELECT * EXCLUDE() for staging CTAS, (b) generate column mappings via INFORMATION_SCHEMA, (c) use a Jinja/dbt template if scaling to more tables.

P1 — RECOMMENDED

Add Alerting for Consecutive Failures

Create a monitoring query or Snowflake alert that fires when CDC_EXECUTION_LOG shows ≥3 consecutive failures for this table. Integrate with email/Slack notification.

P2 — NICE TO HAVE

Improve Batch ID Uniqueness

Append UUID_STRING() or use SYSTIMESTAMP with fractional seconds to prevent batch ID collisions. E.g., 'BATCH_' || TO_VARCHAR(CURRENT_TIMESTAMP(), 'YYYYMMDD_HH24MISSFF3')

P2 — NICE TO HAVE

Conditional Pre-Merge Duplicate Check

The duplicate detection join (staging vs full Bronze table) can be expensive at scale. Consider making it conditional: only run when v_rows_inserted > 0 to skip the join when there are only updates/deletes.
